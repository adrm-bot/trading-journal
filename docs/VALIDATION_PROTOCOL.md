# HT1 v2 백테스트 검증 프로토콜

> `HT1_v2_strategy.pine`(전략 테스터)와 `HT1_v2_indicator.pine`(지표)의 정합 검증, 트리거 A/B/C 성과 비교, 과최적화 방지를 위한 실행 절차. 근거: [HT1_v2_SPEC.md](HT1_v2_SPEC.md) §15.
> ⚠️ 본 문서의 모든 수치 표는 **기록 양식 예시**이며 실측값이 아니다.

## 1. 데이터 준비

### 1.1 TradingView 봉 로딩 특성 (중요 — 비교 무효의 1순위 원인)
- 심볼 첫 진입 시 차트는 **초기 ~2만 봉만** 로드된다 (5m 기준 ~73일). 차트를 **왼쪽으로 스크롤**하면 과거 봉이 추가 로드되고 전략 테스터 기간도 늘어난다.
- **A/B/C 비교는 반드시 같은 로드 상태에서**: 한 모드 실행 후 **새로고침 없이** 설정만 바꿔 재실행. 새로고침하면 로드량이 달라져 비교가 무효가 된다.
- 회차 시작 체크: `□ 심볼 □ TF □ 테스터 기간(시작~끝) □ 직전 회차와 기간 동일 여부`

### 1.2 표본 최소 기준
- 단일 심볼·TF에서 **트레이드 ≥ 100건** 목표. 미달 시: ① 과거 봉 추가 로드, ② 멀티심볼 합산 (BTC/ETH/SOL/BNB/XRP — USDT.P 5종), ③ 15m+1h 멀티TF 합산.
- 100건 미만 표본으로 트리거 모드를 확정하지 말 것 — "우세 후보" 표기까지만.

## 2. 지표-전략 정합 게이트 (0단계 — A/B/C 이전 1회 필수)

### 2.1 시그널 봉 일치
지표와 전략을 같은 차트에 올리고, 지표 라벨(`L 78 A`)과 전략 진입 화살표가 **같은 봉**에서 발생하는지 10건 이상 확인. 1건이라도 불일치 → 중단하고 코어 동기화 검증 (README의 PowerShell diff).

### 2.2 체결 모델 분리 (SPEC §15)
1. **회차 ①**: 전략 Properties → "Fill orders on bar close" **ON** → 지표 집계(E[R]·승률)와 비교. 차이 = 동시터치 모호성 처리 차이 (Bar Magnifier가 정밀 판정).
2. **회차 ②**: **OFF** (기본, 다음 봉 시가 체결 = 실거래 정합) → ①과의 PnL 차이 = **순수 지연·슬리피지 비용**. 이 수치를 기록해 두면 실거래 비용 추정치가 된다.

### 2.3 불일치 시 디버깅 순서
거래 수 불일치 → ① **margin 거절** (전략 선언부 margin_long/short=10 확인 — Pine v6 기본 100은 1x 초과 주문 거절) ② 포지션 중첩(전략은 동시 1포지션, 지표 트래커는 전 시그널 추적 — 일부 차이는 정상) ③ qty 0/na ④ 수수료·슬리피지 설정.

## 3. 트리거 A/B/C 실험 설계

### 3.1 통제 규칙
- 동일 심볼·TF·데이터 범위·프리셋(표준) 고정, **trig_mode만 변경** (breakout / pattern / macd).
- 차트 새로고침 금지. 설정 변경 → 테스터 재계산 완료 대기 → 기록.

### 3.2 기록 양식 (회차당)
```markdown
| 회차 | 심볼 | TF | 트리거 | 기간 | 거래수 | PnL% | MDD% | PF | 승률 | E[R] | MAR* | R/일 | 모호율 | 만료율 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| 1 | BTCUSDT.P | 15m | breakout | … | … | … | … | … | … | … | … | … | … | … |
```
\* MAR = 연환산 수익률 ÷ MDD. 연환산 = PnL% × (365 ÷ 기간일수). R/일 = 총 실현 R ÷ 기간일수.

### 3.3 판정 기준
- **1차**: MAR (높을수록 우위), R/일 ≥ 0.1R 선호. 모호율 ≤10%, 만료율 ≤30% (위반 시 해당 회차 신뢰도 하향).
- **2차**: E[R], PF ≥ 1.3, 승률, 월 거래 빈도 ≥ 5건.
- **동률 타이브레이커**: PF → R/일 → 프리셋 강건성(아래 §4.1).
- 진단: 모호율 >10% = TF 과대 → 하위 TF 재시도. 만료율 >30% = eval 60봉 부족 또는 추세 부재 구간.

## 4. 과최적화 가드

### 4.1 프리셋 plateau (트리거 확정 후)
선정 트리거로 보수/표준/공격 3회 실행. **양 극단 MAR이 표준의 70% 이상**이면 plateau 통과 = 곡선맞춤 아님. 한 프리셋만 튀면 그 성과는 신뢰하지 말 것.

### 4.2 멀티심볼 일반화
같은 설정으로 5심볼(BTC/ETH/SOL/BNB/XRP USDT.P) × 15m/1h. **4/5 심볼 이상에서 E[R] > 0** 이면 일반화 합격. 특정 심볼만 좋으면 그 심볼 특화가 아니라 우연일 가능성을 먼저 의심.

### 4.3 워크포워드 (장기 운영 단계)
IS 6개월 → OOS 2개월 롤링. OOS 성과가 IS의 70% 이상이면 유지. 50% 미만이면 **파라미터를 다시 만지지 말고** 시장 레짐 변화를 먼저 분석.

### 4.4 금지 규칙
- ❌ 결과를 본 후 마음에 안 들어 입력값을 바꿔 재실행 → 그 결과로 보고 (사후 선택 편향)
- ❌ 성과 나쁜 심볼만 제외하고 합산 보고
- ✓ 새 가설은 항상 **다음 독립 회차**에서 검증

## 5. 결과 보관
회차별 결과를 `docs/VALIDATION_RESULTS/{YYYYMM}_{심볼}_{TF}_{트리거}.md`로 저장 (§3.2 양식). 등급(A/B/C)별 분해는 기록하되, 합격 판정은 **전체 거래 합산** 기준 (등급별 라우팅은 Phase C에서 활용).

## 6. 실행 전 체크리스트
```
□ 코어 동기화 diff 통과
□ margin_long/short=10 확인 (v6 기본값 함정)
□ 심볼 = 무기한(.P), 데이터 범위 고정·기록
□ 정합 게이트(§2) 1회 통과
□ A/B/C 동일 범위 3회 + 기록
□ 프리셋 plateau → 멀티심볼 → (운영기) 워크포워드
□ 결과 후 파라미터 변경 금지
```
