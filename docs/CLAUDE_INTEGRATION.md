# TradingView ↔ Claude 연계 시스템 설계서

작성: 2026-06 | 대상: HT1 v2 (크립토 USDT 무기한선물) | 최종 목표: **거래소 API 완전 자동매매**
환경: Windows 11 + Claude Code(데스크톱) + Slack MCP + Claude in Chrome + 스케줄 태스크 | TV 플랜: Premium

---

## 0. 설계 원칙 (리서치·감사 결론의 강제 조항)

1. **신호 채널은 공식 webhook 단일화.** TradingView는 retail용 데이터 API가 없다(공식 확인). 지표 값을 밖으로 빼는 모든 비공식 경로(웹소켓 스크레이퍼, 비공식 MCP)는 ToS 위반·계정 차단·무통보 파손 리스크 → **리서치 보조로만 격리**, 주문 경로 사용 금지.
2. **Claude는 거부권(veto)과 리포트만, 집행은 결정적 룰 엔진.** LLM은 비결정적 → 백테스트 불가. 진입 "생성"이 아닌 "거부/축소"만 맡긴다. Anthropic 금융 가이드라인(human-in-the-loop 권장)과도 정합.
3. **알림 페이로드 정본은 [alert_webhook_schema.json](alert_webhook_schema.json) 하나.** 지표·서버·필터·집행이 전부 이 파일을 참조한다. 멱등키(`id`)는 시그널 봉 기준.
4. **인증 모델 (Pine은 HMAC 계산 불가 — 해시 함수 없음):**
   TV 발신 IP 4개 화이트리스트(`52.89.214.238`, `34.212.75.30`, `54.218.53.128`, `52.32.178.7`) + **URL 경로 시크릿**(`/hook/{secret}`) + 페이로드 `ts` 신선도(±2분) + 멱등키 중복 차단. 시크릿 로테이션 = TV 알림 URL 일괄 교체.
5. **webhook 운영 제약 (공식):** 수신 서버는 **3초 내 응답** 필수 → 즉시 202 반환 후 비동기 처리. 전달 보장 없음 → 멱등성 + 포지션 reconcile 필수. 3분 내 15회 초과 발화 시 알림 자동 정지. 메시지 최대 40,960자.

## 2. 단계별 로드맵

| Phase | 목표 | 난이도 | 비용 | 소요 | 상태 |
|---|---|---|---|---|---|
| **A** | 지표 v2 + JSON 알림 표준화 | 중 | 0원 | — | **본 세션 완료** (pine/, docs/) |
| **B** | 웹훅 수신 + Claude veto | 중 | 개발기 0원(로컬+cloudflared) → VPS $5~10/월 + API $10~30/월 | 1주 | 다음 세션 |
| **C** | 집행 계층 (반자동→자동) | 중상 | B에 포함 | 2~3주 + 운영 4주 | 이후 |
| **D** | 피드백 루프 (저널+리포트) | 중 | API 소폭 | 1~2주 | 이후 |

### Phase B — 수신 서버 + Claude veto
- **인프라**: 개발·검증은 로컬 Windows + cloudflared 터널(0원, 실주문 없음). 실주문 단계(C-1)부터 VPS 이전 — ADR-001.
- **구성**: `server/listener.py`(FastAPI: 202 즉시 응답·검증·멱등키·SQLite 기록) → `server/enricher.py`(ccxt: 펀딩비·OI 변화율·4h/1D 구조 스냅샷) → `llm/signal_filter.py`(Claude API).
- **Claude 프롬프트 구조**: 시스템("백테스트된 룰의 거부권 심사관. 진입 제안 금지") + 전략 룰 요약(프롬프트 캐싱) + 동적 컨텍스트 JSON + 시그널 JSON(ctx 포함).
- **출력 스키마** (structured output, temperature 0):
```json
{"decision":"approve|reject|reduce","size_mult":1.0,
 "checks":{"funding_extreme":false,"oi_divergence":false,"htf_conflict":false,
           "recent_streak_risk":false,"vol_anomaly":false},
 "reasons":["..."],"confidence":0.8}
```
- **Fail policy**: API 5초 타임아웃/장애 시 — 반자동 단계 "미판정으로 Slack 전달", 자동 단계 "reject"(시그널은 또 온다).
- **등급 라우팅**: 지표는 서버 도입 후 **전 등급(C 포함) 송출**로 전환 → 서버가 A/B만 집행 후보, C는 섀도우 추적 전용. (C를 지표에서 차단하면 veto 적중률·등급 임계 타당성을 영구 검증 불가)

### Phase C — 집행 (ADR-002: 반자동 선행)
- **C-1 반자동**: ④ 통과 시그널을 Slack 카드로 발송(판정·사유·체결 예정가) → ✅/❌ 이모지 승인 → 집행. 승인 타임아웃 = 1봉(만료 시 자동 폐기 — 눌림목은 신선도가 생명).
- **C-2 자동 전환 조건** (4주+ & 30시그널+ 운영 후): (a) 수동 거부가 성과를 개선하지 못했음이 데이터로 확인, (b) 리스크 가드 모의 장애 훈련 통과, (c) reconcile 루프 검증. 전환 후에도 파라미터 변경 직후 2주는 반자동 복귀.
- **리스크 가드 (④, 코드 강제)**: 일손실 −3R 도달 시 당일 차단 · 최대 동시 포지션 2 · 심볼당 1 · 지정가 IOC + 슬리피지 상한 0.1% · 시그널 entry 대비 현재가 괴리 > 0.5×risk_atr×ATR이면 스킵(지표 갭 가드와 동일 임계) · **kill switch 3중**(Slack `!halt`/`!flat` + 서버 파일 플래그 + API 키 출금권한 제거).
- **수량 권위**: 페이로드 `qty_hint`는 참고치. 서버가 실잔고 + 거래소 instrument info(qtyStep·minOrderQty) + **추정 강제청산가가 SL보다 불리한 위치인지 검증** 후 재계산.
- **모든 주문에 `clientOrderId = 멱등키`** → 재시작·중복 웹훅에도 멱등. 진입 직후 거래소 네이티브 SL/TP(reduce-only) 동시 등록 — 서버가 죽어도 손절은 거래소가 집행.

### Phase D — 피드백 루프
- journal DB: 시그널 원문(ctx 포함), Claude 판정 전문, 승인 여부, 체결가, 실현 R, 펀딩 비용, **지표 이론가 vs 실체결가 괴리**(E[Δ/R] 실측).
- Claude Code 스케줄 태스크(매일 아침): DB 동기화 → 일일 리포트(승률·R 분포·**veto 적중률** = 거절 시그널의 가상 성과 vs 승인 시그널 성과) → Slack.
- 주간: 파라미터 plateau 분석. **제안은 항상 "다음 워크포워드 구간에서 검증할 가설"로만** — 즉시 적용 금지(과최적화 가드).

## 3. 핵심 ADR

### ADR-001: 웹훅 수신 인프라 — 경량 VPS (개발기는 로컬+터널)
| 옵션 | 고정 IP (거래소 키 IP 제한) | 3초 룰 | 상시 WebSocket reconcile | 비용 |
|---|---|---|---|---|
| **경량 VPS** ✅ | **가능** | 문제없음 | **가능** | $5~10/월 |
| Cloudflare Workers | 불가 | 우수 | 불가 | 무료 |
| 로컬 PC+터널 | 불가 | 양호 | 재부팅·절전 취약 | 0원 |

결정 근거: ① 거래소 API 키 IP 화이트리스트는 보안상 비협상, ② 웹훅 유실 보완용 포지션 상시 감시는 상시 프로세스 필요. Bybit 네이티브 Webhook Signal Trading은 Claude 필터·리스크 가드를 끼울 수 없어 fallback으로만 보존.

### ADR-002: 반자동 선행, 데이터 기반 자동 전환 (위 Phase C 참조)

### ADR-003: LLM 판정 신뢰성 4중 장치
1. **권한 제한**: reject/reduce만. approve = "거부 사유 없음". 진입·가격·방향 생성 권한 없음. 룰 엔진이 LLM 출력과 무관하게 한도 강제.
2. **구조화 출력**: 고정 JSON 스키마 + temperature 0 + 체크리스트형(`checks`) 판정. 스키마 위반 = reject.
3. **섀도우 평가**: 거절 시그널도 가상 체결 추적 → veto 적중률을 분기마다 측정, 기여가 음수면 필터 제거. LLM 필터 자체가 성과 지표로 관리됨.
4. **재현성 로그**: 호출별 프롬프트 전문·모델 버전·컨텍스트 스냅샷 DB 보존. 프롬프트 변경은 버전 태깅 후 성과 구간 분리.

## 5. 최소 PoC (1주, 주문 없음 — 다음 세션 후보)

목표: **"TV 알림 → Claude 판정 → Slack 카드"가 실데이터로 도는 것** (비용: API 몇 달러).
- D1: HT1 v2 알림을 BTC/ETH/SOL 15m에 설정 (Phase A 완료물 그대로)
- D2~3: 로컬 + cloudflared + FastAPI 수신 (202·검증·멱등키·SQLite)
- D4~5: ccxt 펀딩·OI 수집 + Claude veto 호출 연결
- D6: Slack 판정 카드 발송 (승인 버튼 무동작)
- D7: 시그널 5~10건 관찰 — 판정 품질·지연(<5초)·페이로드 정합 검토
모듈(listener/enricher/filter)을 처음부터 분리해 VPS 이전 시 코드 재사용 100%.

## 6. 운영 fail-safe 요약

- 웹훅 유실: WebSocket 포지션 감시가 DB↔거래소 포지션을 60초마다 reconcile, 불일치 시 Slack 경보
- 서버 다운: 거래소 네이티브 SL/TP가 이미 등록되어 있음 + systemd 재시작 + 외부 heartbeat(10분 무응답 시 휴대폰 경보)
- Claude API 장애: fail policy (반자동=미판정 전달 / 자동=reject)
- API 키: 거래 전용(출금 제거) + IP 화이트리스트 + VPS secrets(600) 보관, 로컬 미보관
